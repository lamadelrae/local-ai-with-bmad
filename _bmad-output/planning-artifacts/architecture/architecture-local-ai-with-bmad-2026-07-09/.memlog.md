- (decision) [ADOPTED] AD-2: inference and app are separate processes with independent lifecycles (Docker vs npm run dev). Prevents: embedding an inference runtime in-process.
- (decision) [ADOPTED] AD-4: no Ollama or alternate local-model runner — only the llama.cpp server container. Prevents: a second competing local-runner path.
- (decision) [ADOPTED] AD-5: errors surface plainly (FR-4 status indicator), not engineered around. Prevents: retry/backoff/circuit-breaker logic.

- (decision) Supersedes AD-3 (No persistence layer): conversations, messages, and attachments persist in a single local SQLite file, written directly by the Next.js server process (no new server/service, no Docker volume) — keeps AD-2's process boundary intact. Binds: SQLite file lives in the app's local filesystem, not in the llama.cpp container. Prevents: adding a database server process.
- (decision) Attachments (images, rendered PDF pages) are stored as BLOBs inside the same SQLite database, not on a separate filesystem path. Binds: one physical file is the entire persistence layer. Prevents: a second on-disk attachment store the single-file NFR would have to account for. Trade-off accepted: DB file grows with attachment volume; user chose single-file purity over read/write efficiency at this hobby scale.
- (decision) SQLite accessed via Drizzle ORM + better-sqlite3 (typed schema, lightweight migrations), not raw SQL. Binds: schema changes go through Drizzle migrations. Prevents: hand-written, undocumented schema drift as the conversation/message/attachment tables evolve.
- (decision) Multimodal input (images, audio) rides the same llama.cpp container as text chat, via mtmd/mmproj through the existing OpenAI-compatible endpoint (image_url and input_audio content parts). Extends AD-4's one-model spirit: no second vision or STT model/container is introduced. Binds: FR-6 and FR-7 are additive content-part types on the existing /api/chat contract (AD-1), not new inference services.
- (decision) PDF attachments (FR-6) are rendered to page images client- or server-side before being sent as image_url multimodal parts — no text-extraction/OCR fallback. Binds: a PDF-to-image rendering dependency is introduced (mechanism TBD at build time — pdf.js/pdf-to-img class library). Prevents: a parallel text-extraction code path. Practical limit: only a bounded number of leading pages are rendered/sent (exact cap an open question below).
- (decision) Voice-to-text (FR-7) is a dedicated /api/transcribe route: browser MediaRecorder captures one bounded recording (not live/streaming), sends it as input_audio to the same Gemma-4-E2B container with a transcription-focused system prompt, response text populates the message input box for user review before send. Binds: single-shot record-then-transcribe flow. Prevents: adding a second STT model/container; prevents building live/streaming transcription (deferred, see below).
- (question) What's the practical cap on PDF pages rendered/sent per attachment, and on recording length for voice-to-text, given llama.cpp mtmd's ~30s audio chunking and per-request payload size for multiple page images? Needs a concrete number at build/implementation time, not left to 'reasonable.'
